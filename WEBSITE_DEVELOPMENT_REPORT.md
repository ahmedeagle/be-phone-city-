## 🔴 القسم الأول: إصلاحات حرجة (يجب تنفيذها فوراً)

هذه مشاكل موجودة حالياً وتؤثر على تجربة المستخدم أو عملية البيع.

### 1. ❌ نظام كود الخصم — الشروط لا تعمل
| البند | التفاصيل |
|-------|----------|
| **المشكلة** | الأدمن يضيف أكواد خصم مع شروط (أول طلب، حد أدنى للمبلغ، حد أدنى للكمية، عميل جديد) لكن الشروط **لا يتم التحقق منها** أثناء الشراء — أي شخص يكتب الكود يحصل على الخصم بدون تحقق |
| **السبب الجذري** | `DiscountService::processDiscount()` كان يتحقق فقط من صلاحية الكود والتاريخ، لكن **لا يتحقق من حقل `condition` الـ JSON** |
| **الحالة** | ✅ **تم الإصلاح** — أضفنا `validateConditions()` في `DiscountService` يتحقق من: `first_order`, `min_amount`, `min_quantity`, `new_customer` |
| **الملفات المعدلة** | `DiscountService.php`, `OrderCalculationService.php`, `OrderController.php`, `DiscountController.php` |

### 2. ❌ واجهة كود الخصم كانت معطلة في الـ Frontend
| البند | التفاصيل |
|-------|----------|
| **المشكلة** | كود HTML الخاص بإدخال كود الخصم في صفحة الشراء كان **محذوف/معلق (commented out)** بالكامل |
| **الحالة** | ✅ **تم الإصلاح** — أعدنا تفعيل واجهة إدخال الكود مع ربطها بـ API التحقق |
| **الملفات المعدلة** | `Ordersummary.tsx`, `CheckoutSummarySection.tsx` |

### 3. ❌ قوالب الإيميل تظهر اسم الأدمن الشخصي بدلاً من اسم الموقع
| البند | التفاصيل |
|-------|----------|
| **المشكلة** | عند إرسال إشعارات للأدمن (طلب جديد، تقييم جديد، تذكرة دعم..) يظهر "مرحباً [اسم الأدمن الشخصي]" بدل "مرحباً City Phones" |
| **السبب الجذري** | جميع الـ Notifications تستخدم `$notifiable->name` — عندما يكون المستلم Admin يظهر اسمه الشخصي |
| **الحالة** | ✅ **تم الإصلاح** — أصبح يستخدم `config('app.name')` للأدمن |
| **الملفات المعدلة** | `OrderNotification.php`, `TicketNotification.php`, `PaymentProofNotification.php`, `ReviewNotification.php`, `ContactRequestNotification.php` |

### 4. ⚠️ تفاصيل الطلب لا تعرض تفاصيل الأسعار الكاملة
| البند | التفاصيل |
|-------|----------|
| **المشكلة** | في صفحة تفاصيل الطلب، المنتجات تعرض السعر النهائي فقط — **لا يظهر السعر الأصلي ولا نسبة/مبلغ الخصم** |
| **السبب الجذري** | `ProductResource::forOrderItem()` و `ProductOptionResource::forOrderItem()` لا يرجعان حقول الأسعار (فقط id, name, images) |
| **الأولوية** | متوسطة — المعلومة مفيدة لكن لا تمنع الشراء |
| **الحالة** | 🔄 **قيد التنفيذ** |

### 5. 🟡 وسائل الدفع الإلكتروني — التأكيد التلقائي
| البند | التفاصيل |
|-------|----------|
| **الحالة الحالية** | ✅ **النظام يعمل بشكل صحيح** — عند نجاح الدفع عبر Tabby/Tamara/Amwal/Moyasar: |
| | • `payment_status` يتحول تلقائياً إلى `paid` |
| | • المخزون ينخفض تلقائياً |
| | • إشعار يُرسل للعميل |
| | • نقاط الولاء تُضاف تلقائياً |
| | • شحنة OTO تُنشأ تلقائياً (للتوصيل المنزلي) |
| | • حالة الطلب تتحول إلى `processing` |
| **التحويل البنكي** | ✅ يحتاج موافقة الأدمن (صحيح) — الأدمن يراجع إثبات التحويل ويوافق أو يرفض |
| **ملاحظة مهمة** | ⚠️ مصطلح "confirmed" كحالة طلب لا يُستخدم تلقائياً — الطلب يقفز من `pending` مباشرة إلى `processing` عند نجاح الدفع. لو مطلوب مرحلة "تأكيد" صريحة قبل المعالجة، يحتاج تطوير |
| **تحسين مقترح** | إضافة خيار في الإعدادات للأدمن يتحكم من خلاله: هل الطلبات المدفوعة إلكترونياً تتأكد تلقائياً أم تحتاج مراجعة يدوية أولاً |

## � تحليل تدفق الشحن والتوصيل (Shipping Flow Audit)

### المخطط الكامل لدورة حياة الشحنة

```
إنشاء الطلب ← الدفع ← إنشاء شحنة OTO تلقائياً ← الاستلام من المخزن ← في الطريق ← خارج للتوصيل ← التسليم/الفشل
    │              │              │                         │               │              │             │
 pending       paid          processing                 shipped       in_progress    delivered    returned/failed
    │              │              │                         │               │              │             │
    │              │              │                         │               │              │             │
  لا إشعار    إشعار "طلب     إشعار "حالة           إشعار "حالة      إشعار "حالة    إشعار "حالة     ❌ لا إشعار
              جديد" للعميل   الطلب تغيّرت"          الطلب تغيّرت"    الطلب تغيّرت"  الطلب تغيّرت"   (فجوة!)
              + الأدمن       للعميل فقط             للعميل فقط       للعميل فقط     للعميل فقط
                                                                                    + طلب تقييم
```

### ✅ ما يعمل بشكل جيد في تدفق الشحن

| المرحلة | الآلية | الحالة |
|---------|--------|--------|
| حساب تكلفة الشحن | `ShippingService` — رسوم حسب المدينة + حد الشحن المجاني | ✅ يعمل |
| إنشاء شحنة OTO تلقائياً | `PaymentService::handleAutomaticOtoShipping()` — بعد نجاح الدفع مباشرة | ✅ يعمل |
| مزامنة حالة الشحن | `OtoShipmentWebhookController` — يستقبل تحديثات OTO تلقائياً | ✅ يعمل |
| تحقق من التوقيع | `X-OTO-Signature` — HMAC-SHA256 | ✅ آمن |
| ربط حالات OTO بحالات الطلب | `OtoStatusMapper` — 15+ حالة مربوطة | ✅ شامل |
| إنشاء شحنة يدوي | أدمن يختار شركة الشحن من الخيارات المتاحة | ✅ يعمل |
| مزامنة يدوية من الأدمن | زر "مزامنة حالة OTO" في لوحة التحكم | ✅ يعمل |
| إلغاء الشحنة | إلغاء من الأدمن عبر OTO API | ✅ يعمل |
| عرض رابط التتبع | الأدمن يقدر يفتح رابط شركة الشحن | ✅ يعمل |
| استرجاع المخزون عند الإلغاء | `OrderObserver` — يعيد المخزون تلقائياً | ✅ يعمل |

### تفاصيل إجراءات الأدمن في لوحة التحكم (Filament)

| الإجراء | متى يظهر | ماذا يفعل |
|---------|---------|----------|
| **إنشاء طلب OTO** | الطلب مؤهل للشحن + لا يوجد `oto_order_id` | ينشئ طلب في نظام OTO (بدون شحنة بعد) |
| **إنشاء شحنة** | يوجد `oto_order_id` + لا يوجد `tracking_number` | يعرض شركات الشحن المتاحة (الاسم، السعر، الوقت المتوقع) والأدمن يختار |
| **عرض التتبع** | يوجد `tracking_url` | يفتح رابط تتبع شركة الشحن في تبويب جديد |
| **مزامنة حالة OTO** | يوجد `tracking_number` أو `oto_order_id` | يستعلم من OTO API ويحدث الحالة محلياً |
| **إلغاء الطلب/الشحنة** | يوجد شحنة + الحالة ≠ ملغي | يلغي الشحنة في OTO + يلغي الطلب محلياً |
| **قبول الدفع** | `payment_status = awaiting_review` (تحويل بنكي) | يوافق على إثبات التحويل → يفعّل الشحن التلقائي |
| **رفض الدفع** | `payment_status = awaiting_review` | يرفض إثبات التحويل |

### 🔴 فجوات حرجة في تدفق الشحن

#### الفجوة 1: ❌ العميل لا يرى معلومات التتبع
| البند | التفاصيل |
|-------|----------|
| **المشكلة** | الـ Backend يحفظ: `tracking_number`, `tracking_url`, `shipping_provider`, `shipping_eta`, `tracking_status` ومصفوفة `events` — لكن **لا شيء من هذا يظهر للعميل في الـ Frontend** |
| **التأثير** | العميل يرى فقط "قيد المعالجة" أو "تم الشحن" بدون أي تفاصيل — لا يعرف رقم الشحنة ولا يقدر يتتبع |
| **الإصلاح المطلوب** | إضافة بطاقة تتبع في صفحة تفاصيل الطلب تعرض: |
| | • **رقم الشحنة** (clickable) |
| | • **رابط التتبع** لدى شركة الشحن |
| | • **الموعد المتوقع للتسليم** (ETA) |
| | • **Timeline مرئي** يعرض مراحل الشحنة (أُنشئت → استُلمت → في الطريق → خارج للتوصيل → تم التسليم) |
| **الأولوية** | 🔴 **حرجة** — من أهم المعلومات للعميل |

#### الفجوة 2: ❌ لا يوجد إشعار عند فشل التوصيل أو الإرجاع
| البند | التفاصيل |
|-------|----------|
| **المشكلة** | عندما يفشل التوصيل (`failed`, `returned`, `return_to_sender`) — الـ `OtoStatusMapper` يرجع `null` (لا يغيّر حالة الطلب) |
| **السبب** | بما أن حالة الطلب لا تتغيّر، `OrderObserver` لا يطلق أي إشعار |
| **النتيجة** | ❌ العميل لا يعرف أن شحنته فشلت أو رُجعت |
| | ❌ الأدمن لا يحصل على إشعار بفشل التوصيل |
| | ⚠️ `tracking_status` يتحدث في قاعدة البيانات لكن **لا أحد يُبلّغ** |
| **الإصلاح المطلوب** | |
| | 1. إضافة إشعار خاص لفشل التوصيل/الإرجاع يُرسل للعميل والأدمن |
| | 2. إرسال إيميل: "نأسف، لم نتمكن من توصيل طلبك — سيتم التواصل معك" |
| | 3. إشعار Filament للأدمن: "⚠️ فشل توصيل الطلب #XXXX — يحتاج متابعة" |
| **الأولوية** | 🔴 **حرجة** — العميل يجب أن يعرف |

#### الفجوة 3: ❌ فشل إنشاء الشحنة التلقائي صامت
| البند | التفاصيل |
|-------|----------|
| **المشكلة** | إذا فشل الاتصال بـ OTO API أثناء إنشاء الشحنة التلقائية (بعد الدفع)، الخطأ يُسجّل في الـ Log فقط |
| **النتيجة** | ❌ الطلب مدفوع لكن لا توجد شحنة |
| | ❌ لا يوجد إشعار للأدمن بالفشل |
| | ❌ لا توجد آلية إعادة محاولة (Retry Queue) |
| **الإصلاح المطلوب** | |
| | 1. إشعار فوري للأدمن: "⚠️ فشل إنشاء شحنة للطلب #XXXX — يحتاج إنشاء يدوي" |
| | 2. إضافة Queue Job لإعادة المحاولة بعد 5/15/30 دقيقة |
| | 3. إضافة فلتر في لوحة التحكم: "طلبات مدفوعة بدون شحنة" |
| **الأولوية** | 🔴 **حرجة** — طلبات قد تضيع |

| البند | التفاصيل |
|-------|----------|
| **المشكلة** | لا يوجد زر "تحديث حالة الشحنة" للعميل — يعتمد على Webhooks فقط |
| **التأثير** | إذا تأخر الـ Webhook، العميل يرى معلومات قديمة |

### 🔴 الفجوات الحرجة في نظام الإشعارات

#### 1. ❌ لا يوجد إشعار لفشل التوصيل / الإرجاع
- عندما OTO يرسل `failed` أو `returned` أو `return_to_sender`:
  - `OtoStatusMapper::mapToOrderStatus()` يرجع `null`
  - حالة الطلب **لا تتغيّر**
  - `OrderObserver` لا يطلق `notifyOrderStatusChanged()`
  - **النتيجة: صمت تام — لا العميل ولا الأدمن يعرف**
- ⚠️ `tracking_status` يتحدث في قاعدة البيانات لكن لا أحد يتابعه

#### 2. ❌ لا يوجد إشعار لمحاولة التوصيل الفاشلة
- بعض شركات الشحن تحاول التوصيل أكثر من مرة
- لا يوجد إشعار "لم نجدك في المنزل — سنحاول مرة أخرى غداً"

#### 3. ❌ الأدمن لا يحصل على إشعار لتغييرات حالة الطلب
- فقط `notifyOrderCreated()` يرسل للأدمن
- `notifyOrderStatusChanged()` يرسل **للعميل فقط**
- الأدمن لا يعرف أن طلب تم شحنه أو تسليمه إلا لو فتح لوحة التحكم

#### 4. ❌ لا يوجد إشعار عندما شحنة تتأخر
- لا توجد آلية مراقبة: "الشحنة تجاوزت ETA ولم تُسلّم"
- لا Cron Job يفحص الشحنات المتأخرة

#### 5. ❌ لا يوجد إشعار Push / Real-time
- جميع الإشعارات عبر mail + database فقط
- لا يوجد WebSocket أو Push Notification للتحديثات الفورية
